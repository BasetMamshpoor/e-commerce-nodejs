# E-Commerce Backend

## وضعیت فعلی
این پروژه مرحله‌به‌مرحله نوشته می‌شود. تا این لحظه آماده است:

- اسکلت کامل پروژه (config، error handling، middlewareهای پایه)
- ماژول کامل **احراز هویت**: ثبت‌نام + OTP، ورود با رمز، ورود با OTP،
  refresh token (با rotation)، logout / logout-all، فراموشی/بازنشانی رمز
- ماژول کامل **دسته‌بندی، برند، ویژگی و محصول** (آیتم‌های ۱، ۲، ۳، ۲۱):
  - دسته‌بندی چندلایه با جلوگیری از حلقه (cycle) و دسترسی به زیرمجموعه‌ها
  - برند و ویژگی (رنگ/جنس/سایز/...) با مقادیر مرتبط
  - محصولات متغیر: هر تنوع با SKU، قیمت، موجودی و تخفیف مجزا
  - فیلدهای denormalize شده (minPrice/maxPrice/isInStock/hasActiveDiscount)
    روی محصول برای فیلتر/مرتب‌سازی سریع در صفحه‌ی فروشگاه
  - لیست محصولات با فیلتر دسته‌بندی (+ زیرمجموعه‌ها)، برند، بازه‌ی قیمت،
    ترکیب ویژگی‌ها، جست‌وجو، در انبار/تخفیف‌دار، مرتب‌سازی، صفحه‌بندی
  - مدیریت تصاویر محصول/تنوع (با ارجاع به Media — ماژول رسانه بعداً اضافه می‌شود)
  - ثبت بازدید محصول (برای آیتم ۱۶ — نمودارهای آنالیزی)
- کنترل‌های امنیتی پایه: قفل حساب بعد از تلاش‌های ناموفق، محدودیت تعداد
  نشست‌های فعال، بلاک IP، rate limiter عمومی
- ماژول کامل **سبد خرید، علاقه‌مندی و مقایسه محصولات** (آیتم‌های ۵، ۶، ۷):
  - سبد خرید مهمان (بدون لاگین) + کاربر عضو با یک هدر مشترک `X-Guest-Token`
  - افزودن/ویرایش تعداد/حذف آیتم با اصلاح خودکار تعداد اگر از موجودی بیشتر باشد
  - `POST /cart/merge` برای ادغام سبد مهمان با سبد کاربر بعد از لاگین
  - محاسبه‌ی قیمت مؤثر هر آیتم (با درنظرگرفتن تخفیف فعال تنوع) در `utils/pricing.ts`
  - لیست علاقه‌مندی (فقط کاربر عضو) با صفحه‌بندی
  - مقایسه‌ی تا ۴ محصول هم‌زمان (هم مهمان هم عضو)، با همان الگوی guest token
- ۳۷ تست (jest + supertest) — منطق خالص + smoke تست روتینگ/اعتبارسنجی/auth

- ماژول کامل **کد تخفیف** (آیتم ۴) با تمام محدودیت‌های درخواستی:
  - درصدی/مبلغ ثابت + سقف تخفیف برای حالت درصدی
  - اختصاص به محصول/دسته‌بندی/کاربر خاص (هرکدام اختیاری و قابل ترکیب)
  - حداقل مبلغ سبد خرید، سقف تعداد کل استفاده، سقف تعداد استفاده هر کاربر
  - بازه‌ی زمانی فعال‌بودن (startsAt/expiresAt)
  - `POST /discount-codes/apply`: اعتبارسنجی و محاسبه‌ی تخفیف روی سبد خرید
    فعلی کاربر/مهمان (بدون مصرف‌کردن کد) — برای پیش‌نمایش لحظه‌ای در سبد
  - `recordDiscountCodeUsage` آماده است تا ماژول سفارش (مرحله‌ی بعد) بعد از
    ثبت قطعی سفارش صدایش بزند و usageCount/تاریخچه را ثبت کند

ماژول‌های بعدی (سفارش، پرداخت، تیکتینگ، رسانه، ...) در ادامه‌ی همین چت
اضافه می‌شوند.

## راه‌اندازی

```bash
npm install

cp .env.example .env
# مقادیر .env را پر کنید (حداقل DATABASE_URL و دو JWT secret)

# ⚠️ مرحله‌ی ضروری: تولید Prisma Client واقعی
# (در sandbox من دسترسی شبکه به سرورهای Prisma مسدود بود، پس این مرحله
#  هنوز روی این پروژه اجرا نشده — حتماً خودتان با اینترنت کامل اجرا کنید)
npx prisma generate

# اعمال مایگریشن روی دیتابیس
npx prisma migrate dev --name init

npm run dev
```

سپس تست سلامت سرور:
```
GET http://localhost:4000/health
```

## نکات مهم معماری

- **Prisma 7**: کلاینت دیگر از `@prisma/client` مستقیم import نمی‌شود؛
  طبق `output` در `prisma/schema.prisma` در مسیر `src/generated/prisma`
  ساخته می‌شود (همان‌جایی که `src/lib/prisma.ts` از آن import می‌کند).
- **Express 5**: هندلرهای async نیازی به try/catch یا asyncHandler ندارند؛
  خودِ روتر Promise reject شده را به errorHandler می‌فرستد.
- **احراز هویت با قابلیت ابطال فوری**: هر access token یک `sid` (شناسه‌ی
  UserSession) دارد. میدلور `authenticate` این نشست را از دیتابیس می‌خواند
  تا logout / logout-all / بلاک‌شدن کاربر همان لحظه اثر کند (نه فقط بعد از
  انقضای توکن).
- **درگاه پرداخت**: فقط اینترفیس (`src/services/payment/payment.types.ts`)
  و یک رجیستری خالی (`payment.factory.ts`) ساخته شده؛ طبق توافق، اتصال به
  درگاه واقعی را خودتان اضافه می‌کنید.
- **OTP**: کانال SMS فعلاً Mock (در کنسول چاپ می‌شود). کانال Email واقعاً
  با nodemailer کار می‌کند اگر SMTP_HOST را در .env پر کنید، وگرنه آن هم
  در کنسول چاپ می‌شود.
- **قیمت/موجودی denormalize شده روی Product** (`minPrice`, `maxPrice`,
  `isInStock`, `hasActiveDiscount`): چون محصول خودش قیمت ندارد (هر تنوع
  قیمت جدا دارد)، مرتب‌سازی/فیلتر بر اساس قیمت در سطح محصول را با این
  فیلدهای کش‌شده انجام می‌دهیم به‌جای aggregate زنده روی رابطه‌ی variants.
  این فیلدها هر بار که تنوع‌ها تغییر کنند با `recomputeProductAggregates`
  بازمحاسبه می‌شوند (`product.service.ts`). نکته: تخفیف‌های زمان‌دار که
  هنوز شروع نشده‌اند با گذشت زمان به‌صورت خودکار «فعال» نمی‌شوند مگر
  variant دوباره ذخیره شود یا یک cron job دوره‌ای این تابع را صدا بزند —
  این محدودیت را در کامنت بالای تابع هم نوشته‌ام.
- **فیلتر چند ویژگی هم‌زمان** (مثلاً رنگ=قرمز و سایز=L): در سطح محصول
  انجام می‌شود، نه تنوع؛ یعنی محصولی که یک تنوع قرمز و یک تنوع دیگر با
  سایز L دارد هم نتیجه می‌شود (نه لزوماً یک تنوع که هم قرمز هم L باشد).
  این دقیقاً رفتار رایج در صفحه‌ی لیست محصولات بیشتر فروشگاه‌هاست؛
  ترکیب دقیق را کاربر در صفحه‌ی خود محصول انتخاب می‌کند.
- **جلوگیری از حذف ناخواسته**: حذف محصول/تنوعی که در سفارش قبلی استفاده
  شده مسدود می‌شود (پیشنهاد: به‌جای حذف، `status=ARCHIVED` یا
  `isActive=false` کنید). حذف دسته‌بندی‌ای که زیرمجموعه دارد هم مسدود می‌شود.
- **هدر مشترک `X-Guest-Token`**: هم سبد خرید و هم لیست مقایسه برای کاربر
  مهمان از همین یک هدر استفاده می‌کنند (`utils/guestToken.ts`). اگر فرانت
  این هدر را نفرستد، بک‌اند یک UUID تازه می‌سازد و در پاسخ
  (`data.guestToken`) برمی‌گرداند؛ فرانت باید آن را ذخیره (مثلاً
  localStorage) و در درخواست‌های بعدی همان را بفرستد. بعد از لاگین،
  `POST /cart/merge` با همین توکن سبد مهمان را با سبد کاربر ادغام می‌کند.
- **کد تخفیف محصول/دسته‌محور**: تخفیف فقط روی همان بخش از سبد که واجد
  شرایط است محاسبه می‌شود، نه کل سبد (مثل «۱۰٪ تخفیف روی جوراب‌ها»).
  `minCartAmount` اما روی کل مبلغ سبد بررسی می‌شود، نه فقط بخش واجدشرایط.
  جزئیات کامل در کامنت بالای `discount-apply.service.ts`.

## ساختار پوشه‌ها
```
src/
  config/       env validation با zod
  lib/          پریزما کلاینت singleton
  middlewares/  error handler, auth, validate, rate limit, blocked ip
  utils/        ApiError, ApiResponse, jwt, hash, otp helpers
  types/        تایپ‌های افزوده به Express.Request
  services/     منطق اصلی (auth, otp, payment, session, login-guard, catalog, shopping)
  validations/  schema های zod ورودی هر route
  controllers/  هندلرهای HTTP
  routes/       تعریف مسیرها (auth, categories, brands, attributes, products, cart, wishlist, comparison)
  app.ts        ساخت اپ اکسپرس
  server.ts     نقطه‌ی ورود (listen + graceful shutdown)
tests/          تست‌های jest + supertest
prisma/
  schema.prisma
```
